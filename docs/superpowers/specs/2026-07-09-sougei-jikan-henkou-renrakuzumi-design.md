## 1. データモデル：シート「送迎連絡台帳」

**保存モデルは追記型（append-only）を推奨**。理由: 社長要件「履歴を残す」に最も素直で、既存「送迎変更ログ」も `appendRow` の追記型（gas:614）と実装パターンが揃う。現在の状態は「キーごとの最新行が勝ち」で導出する。

### 列定義

| # | 列名 | 例 | 由来 | 説明 |
|---|---|---|---|---|
| 1 | 記録日時 | `2026-07-10 08:42:11` | 追加 | この行が台帳に書かれた時刻（サーバー時刻） |
| 2 | 適用日 | `2026-07-15` | 社長指定 | 変更後の送迎時刻が適用される日（YYYY-MM-DD） |
| 3 | 利用者 | `山田花子` | 社長指定 | override の stop.user と一致 |
| 4 | 時間帯 | `午前` | 追加（任意） | 午前/午後の表示補助＝**先頭の変更スロット**を入れる（一覧sheetの目視用）。**キーには含めない**。AM/PM両方に変更がある稀ケースの完全な情報は列11「変更詳細」に持つ |
| 5 | 旧時間 | `09:10` | 社長指定 | 変更前の迎え時刻＝**先頭スロットの旧時間**（sheet目視用）。**A1採用＝GASが保存時に直前override（無ければ曜日ベース）から自動で引く**（sougei無改修）。引けない時のみ空欄。全スロットは列11 |
| 6 | 新時間 | `09:30` | 社長指定 | 変更後の迎え時刻＝**先頭スロットの新時間**（override pick stop.time）。全スロットは列11 |
| 7 | status | `要連絡` / `連絡済み` / `通常化` | 社長指定 | 行の種別。最新行の status がその(適用日,利用者)の現在状態。`通常化`=E1で色OFF時の解除行 |
| 8 | 連絡者 | `佐藤` | 社長指定 | status=連絡済み の行のみ。誰が電話したか（C1＝ボードGAS staff_list 由来） |
| 9 | 連絡日時 | `2026-07-10 08:42:05` | 社長指定 | status=連絡済み の行のみ。いつ電話し終えたか |
| 10 | source | `出勤送迎表` / `送迎時間一覧` | 追加 | 要連絡・通常化行=書き手発、連絡済み行=一覧発。経路の追跡用 |
| 11 | 変更詳細 | `[{"slot":"午前","old":"09:10","new":"09:30"},{"slot":"午後","old":"13:00","new":"13:20"}]` | 追加 | **AM/PMロスレス**。timeChanged のある迎え(pick)を**AM/PM全部**拾った `[{slot,old,new}]` の JSON。連絡モーダルはこれを読んで「AM旧→新／PM旧→新」を並べる。**先勝ちで片方を捨てない**（社長要件：連絡漏れ＝現場事故の防止）。単一スロットでも常に1要素で入れる |

キー = **(適用日, 利用者)**。**現在状態 = そのキーの最新（記録日時が一番新しい）行の status**。

**AM/PM の扱い（社長確定 2026-07-09）**: キー・表示・連絡アクションは (適用日,利用者) の**1件**（1日1利用者=迎え1連絡）。ただし同一日・同一利用者が AM と PM 両方の迎えで timeChanged を持つ稀ケースで**片方を見落とすと連絡漏れ**になるため、列11「変更詳細」に**AM/PM 全スロットの old→new を残す**（先勝ちで捨てない）。旧時間/新時間/時間帯(列4-6)は先頭スロットの値で sheet 目視を助けるだけ。
**要確認（clasp窓・Task 6の count 関数）**: 実 overrides で「同一日・同一 user が am/pm 両 pick に出る」ケースが 0 件なら、この畳みは落ちる情報ゼロで完全に安全と確定できる（`_count_amPmPickOverlap()` を実行して報告）。

### 1行の具体例（追記型・時系列）

```
記録日時               適用日       利用者    時間帯  旧時間  新時間  status    連絡者  連絡日時               source
2026-07-10 08:40:00   2026-07-15  山田花子  午前    09:10   09:30   要連絡    —      —                     出勤送迎表
2026-07-10 08:42:05   2026-07-15  山田花子  午前    09:10   09:30   連絡済み  佐藤    2026-07-10 08:42:05   送迎時間一覧
```

→ このキーの最新行は 2行目 `連絡済み`＝一覧では色B。1行目（要連絡）は履歴として残る。

もし社長が同じ人をもう一度変更したら 3行目に新しい `要連絡`（新時間更新）を追記＝最新が要連絡に戻り色Aへ。過去の連絡済み履歴は消えない。

## 2. 画面と操作の流れ

```
① 社長が出勤＆送迎表(sougei)で山田さんの迎え時刻を 09:10→09:30 に変更し、
   その stop の「変更」色をON（既存 timeChanged=true）
        │  2秒デバウンスで自動POST saveSchedTimes（overrideDate=2026-07-15 付き）
        ▼
② RELAY GAS が override[2026-07-15] を保存（現行どおり）＋
   【新規】その override の stops から timeChanged=true を拾い、
   台帳に「要連絡」行を1行 upsert相当で追記（status=要連絡, source=出勤送迎表）
        │
        ▼
③ スタッフ用一覧(sched-grid)が getSchedTimes を取得（5分自動 or 手動再読込）。
   override に timeChanged が立ち、かつ台帳の最新が「連絡済み」でない
   → その stop を【色A=要連絡】で表示（「☎ 要連絡」バッジ）
        │
        ▼
④ スタッフが山田さんの家族へ電話し「お迎えが09:30になります」と伝える。
   一覧の「連絡済み」ボタンを押す → 連絡者を選ぶモーダル → 確定
        │  POST markSchedContacted（no-cors）→ §3 のライトバック検証
        ▼
⑤ 台帳に「連絡済み」行を追記（連絡者・連絡日時つき, source=送迎時間一覧）。
   検証OKで一覧は【色B=連絡済み】に確定。
        │
        ▼
⑥ 2026-07-15 を過ぎると override は prune され、一覧はその日を色付けしない＝通常表示。
   台帳の2行（要連絡→連絡済み）は履歴として残る。
```

補足（要件7「確定して公開」廃止・Q2 決定反映）: 調査実測では publish は既に2秒デバウンスの自動同期が実体で、手動「📤送迎時間を共有」ボタンは緊急用（sougei:523 title明記）。**廃止＝この手動ボタンを通常UIから外し、保存＝自動同期を唯一経路にする**という意味で織り込む。台帳の要連絡行も②の自動同期時に即書かれるため「保存＝即反映」と整合する。
- **Q2 決定**: 緊急「共有」ボタンは**初回の本番サイクルだけ隠しフォールバックとして残す**（通常UIからは外すが即撤去はしない）。auto経路（保存＝自動同期で要連絡行が書かれ、E1 の色クリアも回る）が**本番で検証できたら、次フェーズで緊急フォールバックを撤去する（引き算）**。

## 3. 書き込み経路（読み取り専用の一覧から台帳へ安全に書く）

### 3-1. 新エンドポイント
- GAS `doPost` に `action='markSchedContacted'` を追加（設計のみ）。
- 受けるもの: `{ action, date, user, operator, contactedAt }`（キーは (date,user)＝迎えのみso時間帯は不要。表示補助の時間帯は任意で添えてよい）。
- 台帳の該当キー最新が既に連絡済みなら**追記せずOKを返す**（サーバー側 idempotency＝二重押し対策・§4）。

### 3-2. no-cors の罠への対処（**最重要**・メモリ `genba-nocors-post-成否読めない罠`）
sched-grid の書込POSTは genba同様 no-cors になる見込み＝**レスポンスを読めない**。`.then()` は成功を意味せず `.catch()` はサーバーエラーで発火しない。よって成功は自己申告で判断しない。**ライトバック検証**を必須にする:

1. スタッフが「連絡済み」確定 → UIは楽観的に「連絡済み(保留中)」表示（スピナー等）。**この時点では色Bを確定させない**。
2. `markSchedContacted` を no-cors でPOST。
3. `HAICHI_VERIFY_DELAY_MS` 相当（例 4〜6秒）待って、**サーバーを読み直す**（D1採用＝getSchedTimes に台帳現在状態を合成して載せる）。
4. 突合: 読み直した台帳最新が (date,user)=連絡済み なら → 色Bを確定・保留解除。違えば → 色Aへ戻し「記録できませんでした。もう一度お試しください」を表示。
5. 楽観描画は検証結果に関わらず必ず解決する（保留のまま放置しない）。

### 3-3. オリジンガード（メモリ `genba-betsu-origin-touroku-jiko`）
- sched-grid は github.io 配信。**本番オリジン以外（localhost/preview）からの markSchedContacted を先頭ガードで遮断**し、非本番では「連絡済み」ボタンを disabled にする。genba の gnbGuardProdWrite と同型を移植（設計のみ）。
- プレビュー(localhost)は origin ガードで書込後経路を目視できない点も既知（実機github.ioで確認する）。

### 3-4. 連絡者(operator)の取得（C1採用）
- モーダル内に**連絡者セレクタ**を置く（メモリの欠席box/kbox で確立した「担当者はモーダル内選択」パターンに合わせる）。受付者バー直読はしない。
- C1＝**Task 0 調査で確定（2026-07-09）**: 連絡者名簿の正本は **ボードGAS `action=staff_list`**。実体 = `gas/yawaragi-board/コード.js:1684-1687` の分岐 → `getStaffListFromShiftSheet()`（`コード.js:3687-3703`）が **シフト希望SS「スタッフ」シートA列** を文字列配列で返す（`{staff:[...]}`）。リポジトリ全体の「スタッフ名一覧の正本」で、入退社はそのシート更新だけで全アプリ反映。
  - **取得方式**: sched-grid は**素の `fetch`（CORS・JSONPではない）**で通信（`sched-grid.html:546`）。連絡者名簿も `fetch(BOARD_URL + '?action=staff_list&t=' + Date.now()).then(r=>r.json())` → `data.staff`。
  - **URLを1本追加**: sched-grid の既存 `GAS_URL` は**出勤送迎表GAS（V66Udd…）でボードGASではない**。`staff_list` 用に**ボードGASのURLを別途定数追加**する（例 `sougei_nisshi.html:1121` の `AKfycbwo1UGx…` と同一）。
  - **比嘉（社長）**: `data.staff` に社長が含まれない可能性あり（`yawaragi-board.html:5311` 前例）。連絡者に社長を含めるかは Task 9-B で確定（含めるなら先頭固定で補う）。
  - 不採用: 候補2 sougei は静的配列 `STAFF`（`sougei.html:574-591`）で更新にコード改修が要り二重管理。候補3 getStaffMaster は提出物用の役割付きサブ集合（正本は staff_list とコード.js:13401-13404 が明言）。

## 4. エラー対応（「成功」を自己申告しない）

| 事象 | 挙動 |
|---|---|
| 保存失敗（GASエラー/台帳書込失敗） | ライトバック検証(§3-2)が不一致→UIは色Aへ戻す・明示エラー。偽の「連絡済み」を出さない。 |
| 二重押し（同一スタッフ連打） | ①押下中はボタン disabled。②サーバーidempotency＝最新が既に連絡済みなら追記せずOK。二重行を作らない。 |
| 同時押し（別スタッフが同じ利用者） | 追記型＋最新勝ちで連絡済みに収束。エラーにしない。 |
| 通信断（POSTが届いたか不明） | no-cors では判定不能。**ライトバック検証が唯一の裁定者**。検証で連絡済みを確認できるまで色Bにしない。時間切れは「確認できませんでした」を表示し色Aのまま（成功を騙らない）。 |
| 台帳シートが無い | GAS が自動作成（既存 `_ensureChangeLogSheet` gas:575 と同型）。 |
| 旧時間が取得できない | A1で引き当て。直前overrideも曜日ベースも無い等で引けない時は旧時間を空欄で記録し、変更があった事実（新時間・要連絡）は残す。 |

検証の基本姿勢（メモリ `exit0-nonhang-not-functional-proof` / `tool-output-corruption-trap`）: 「固まらず返った＝成功」ではない。確証は既知の一次事実（読み直した台帳値）との突合で取る。

## 5. テスト項目

### 正常系
- [ ] 社長が未来日の迎え時刻を変更＋色ON → 台帳に要連絡行が1行できる → 一覧が色Aで表示。
- [ ] スタッフが連絡済み押下 → ライトバック検証OK → 色B確定＋台帳に連絡済み行（連絡者・連絡日時入り）。
- [ ] 適用日を過ぎる → 一覧はその日を色付けしない（通常）／台帳の履歴2行は残る・検索できる。

### 異常系・境界
- [ ] 同じ利用者を連絡前に2回変更 → 台帳最新が要連絡・新時間が更新・色Aのまま（要連絡は重複増殖しない＝キー最新勝ち）。
- [ ] 連絡済みの後に社長が再変更 → 新しい要連絡行が最新になり色Aへ戻る／過去の連絡済み履歴は残る。
- [ ] 過去日を開く → override無し（prune済）でクラッシュせず通常表示。台帳履歴は残るが色は付けない。
- [ ] 連絡済みの二重押し → 2行目の連絡済みを作らず、偽エラーも出さない（idempotency）。
- [ ] 通信断で連絡済みPOST → ライトバック検証が不一致→色Aへ戻し正直なエラー。偽の連絡済みを出さない。
- [ ] 非本番オリジン(localhost/preview)から連絡済み → オリジンガードで遮断・ボタン disabled。
- [ ] 台帳シート未作成の初回 → GASが自動作成して書込成功。
- [ ] 別スタッフ2人が同じ利用者を同時に連絡済み → 連絡済みに収束・エラーなし。
- [ ] 社長が変更後に色OFF（timeChanged取消）を連絡前に実施 → E1: 台帳に `通常化` 行が追記され最新が通常へ・一覧の色Aが消える。

### 純関数化して Node で回せる範囲（実装フェーズの指針・本設計では対象外）
- 台帳の (適用日,利用者) 最新勝ち導出 / 要連絡・連絡済みの色判定 / idempotency 判定。
