# ①②③ 当月セル色分け（提出可視化）設計書

- 作成: 2026-06-26
- 対象: `個別機能訓練計画書チェック.html`（①）／`oral.html`（②）／`monitoring.html`（③）＋ `shared.js`
- 種別: フロントのみ（GAS無書込・既存GETの読み取りのみ）
- 前提: 横断ビュー④で shared.js §I（正準判定）配信済み。①②③HTMLは本番一致（master `9babc03`）＝stale でない。

## 1. 目的

①個訓・②口腔・③通所の各表で「**今月どのセルを提出するのか**」を色で一目化し、社長が数えなくて済むようにする。

## 2. スコープ

- **当月の「提出セル」のみ**3色化する（他月セルは従来表示のまま＝無回帰）。
- 判定は新規に作らず **shared.js §I の正準関数**を使う。①②③を **§I 参照に切替**（次フェーズ予定の切替を本作業で同時実施）。切替は挙動同一（§Iは各アプリの実装を移植したもの）＝低リスク。
- 3アプリで**色の意味・見た目を統一**。

## 3. 統一3色（状態→色）

「該当」= その月にそのセルの提出義務がある（§Iの当月判定が true）。

| 色 | 状態 | 意味 |
|---|---|---|
| 🔴 赤 | 該当 & **未作成** | 今月これからやる人 |
| 🟢 緑 | 該当 & 作成済 & **未送付** | 提出準備OK（あと送るだけ） |
| 🔵 青 | 該当 & **送付済** | 完了 |

- **該当 = 赤+緑+青**（送付済も「今月該当」に含む）。色付きセル総数 ＝ ①の今月該当人数（isPlanMonth/isHyoukaMonth と同定義）。
- 非該当の当月セル・他月セルは色を付けない。
- 作成ステップが無いセル（③モニタリング送付）は緑が出ず 赤→青 で遷移（3色は語彙であり全セルが3状態を持つ必要はない）。
- **保留/特殊状態は3色を付けず既存表示を維持**（無回帰）：当月でも保留（保険未登録/利用継続未確定 等）のセルは「今月は出せない」ため3色対象外とし、既存の保留色（グレー/オレンジ・⏸/⚠）をそのまま残す。よって突合は「赤+緑+青 ＝ ①該当のうち保留を除いた数」（保留分は別扱い）。

## 4. TDD する純関数（§I に追加・一本化）

色判定を1箇所に集約して TDD する。

```js
// shared.js §I に追加
function submitCellColor(isApplicable, hasCreated, hasSent) {
  if (!isApplicable) return '';        // 非該当＝色なし
  if (hasSent) return 'blue';          // 送付済
  if (hasCreated) return 'green';      // 作成済・未送付
  return 'red';                        // 該当・未作成
}
```
①②③は各々の当月判定（§I）と作成/送付フィールドから `(isApplicable, hasCreated, hasSent)` を作り、この関数の戻り値（red/green/blue）を**統一CSSクラス**に対応付けてセルへ付与する。

統一CSSクラス（①②③に同一値で定義）:
- `.cell-todo`（赤）／`.cell-ready`（緑）／`.cell-done`（青）

## 5. アプリ別マッピング（当月セルのみ）

### ① 個訓（計画セル・評価セル）
- 計画セル：該当=`isPlanMonth(planStart,planMonths,y,m)` ／ 作成=`keikaku_date` ／ 送付=`keikaku_sent_date`
- 評価セル：該当=`isHyoukaMonth(planStart,planMonths,y,m)` ／ 作成=`tasseido_date` ／ 送付=`hyouka_pdf_date || hyouka_print_date`

### ② 口腔（ケアマネ提出セル＝結果報告/評価の送付）
- 該当=`isOralEvalMonth(startedAt,y,m)` ／ 作成=`plan_date` ／ 送付=`sent_to_cm`（真）
- 対象セル＝`renderOralSubmitCell`（モニタリング表のケアマネ提出列）。

### ③ 通所（モニタリング当月送付セル＋計画書カード）
- モニタリング送付（要支援・毎月）：該当=要支援かつ当月 ／ 作成=（なし） ／ 送付=`pdfSendDate || printSendDate`（→赤未送付/青送付済の2状態）
- 最終評価月：`monitoringFinalEvalMonth(planStart,finalEvalMonth)===当月` のセルに ✍（既存）を維持しつつ3色を重ねる
- 計画書カード（当月該当の1年計画）：作成=`plan_date` ／ 送付=`sent_to_cm`

## 6. ①②③ の §I 参照化（同時実施・挙動同一）

- 各HTMLに `<script src="shared.js?v=2026-06-26a"></script>` を本体scriptの前に追加。
- ローカル判定関数を §I 呼び出しへ置換し、重複定義を削除：
  - ①：`isPlanMonth`/`isHyoukaMonth`（§Iと同シグネチャ・同ロジック＝そのまま置換）
  - ②：`isOralEvalMonth`（同上）
  - ③：`getFinalEvalMonth(planStart,override)` → `monitoringFinalEvalMonth(planStart,finalEvalMonth)`（同ロジック。戻り値が `null`→`''` に変わる点のみ注意：③の使用箇所は `=== ymKey` 比較なので `''` でも非一致になり挙動不変。要実測確認）

## 8. 完了条件（実測）

1. ①②③の当月の提出セルが3色（赤/緑/青）で塗られ、色と状態が一致（未作成=赤・作成済未送付=緑・送付済=青）。
2. ①の「今月計画該当／評価該当」人数 ＝ ①の色付きセル総数（赤+緑+青）が一致。
3. get 以外（update*/set*/delete*）が一切飛ばない（②③の表示は読み取りのみ・色分けは表示変更）。
4. genba 無影響（pin により §I 休眠）。
5. §I の `submitCellColor` が node TDD で緑。
6. **無回帰を①②③それぞれで実測**：色が付くのは**当月セルのみ**で、他月セル・既存の状態表示（過去未作成の赤枠・送付済の緑/濃緑・③の✍/final-eval枠・current青枠 等）は**従来どおり**であること。①②③各アプリで目視確認する。
