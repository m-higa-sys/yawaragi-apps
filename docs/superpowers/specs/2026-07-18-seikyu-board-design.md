## 1. 目的・ひとことで

やわらぎの請求管理で、リハブの請求情報CSVを放り込むと「**誰がどの月で入金していないか**」が一目で分かる集計表。社長の主目的は **未入金（赤）の利用者の炙り出し**。

- **本アプリ = 発見の鏡**（read-only・請求時点の俯瞰）
- 追跡・回収の台帳は既存 `furikae`（後述 §9 の住み分け参照）

## 2. スコープ（v1）

- 入力は**リハブ「請求情報（サービス提供年月別）」CSVのみ**。電算結果CSVには依存しない。
- 複数月CSVを一度に放り込み → 縦=利用者・横=月のマトリクス表示 → 未入金を赤で上に炙り出す、まで。
- **手入力（繰越・返金・備考）の永続化は次段**（v1では作らない。§10）。
- **GASなし・単一HTML・POSTゼロ**（本番GAS事故のリスク構造的にゼロ）。ブラウザで開くだけ。

## 3. 入力仕様（実CSVで確認済み・2026-07-18）

令和8年3〜6月の実CSVで二重確認した確定事実。**位置ではなく列名一致（findCol）で拾う**。

- 文字コード: **Shift_JIS**（`TextDecoder('shift-jis')`）。改行は CRLF/NEL 混在 → `\r\n|[\r\n]` で分割。
- 構造: **1行目＝メタ情報（タイトル・出力日時）、2行目＝本ヘッダー（23列）、3行目以降＝データ**。ヘッダーは1行目固定ではない点に注意。
- 使う列（実ヘッダー文字列そのまま）:

| 用途 | 実ヘッダー列名 | 備考 |
|---|---|---|
| 識別キー | `被保険者番号` | 例 `0000100038`（10桁ゼロ埋め） |
| 表示名 | `利用者名` | 例 `伊熊　二三子` |
| 月 | `サービス提供年月` | 例 `202605`。**ファイル名でなくこの列で月判定** |
| 引落し額 | `利用者請求額総額（3+4-5+6+7+8-9）` | メイン数値 |
| 自費(税抜) | `7保険外サービス費（税抜）` | 金額列は数字プレフィックス付き |
| 自費(消費税) | `8保険外サービス費（消費税額）` | 自費合計 = 税抜 + 消費税 |
| 支払方法 | `支払方法` | `口座振替` / `自己負担なし` / `自費負担なし` |
| 入金状況 | `入金状況` | **`入金済` / `未入金` の実文字列**（空欄ではない） |

- 必要列が見つからなければ**明示エラー表示**（想定外フォーマットで黙って壊れない）。

## 4. データ処理パイプライン（純関数で分離＝テスト可能）

1. `decodeCsv(arrayBuffer)` → SJIS→文字列、`\r\n|[\r\n]` で行分割。
2. `parseCsv(text)` → `"..."` 内カンマ/二重引用符エスケープ対応の行配列。
3. `extractRows(rows)`:
   - ヘッダー = `rows[1]`（2行目）を findCol で列名→index 解決。
   - データ = 3行目以降。**除外は値ベース**（行数・件数に一切依存しない）。`利用者名` の値が `総額 / 保険外のみ / 空` のいずれかなら小計行として除外。「先頭N行を捨てる」「9行スキップ」のような**位置・件数ハードコードはしない**。

     ```js
     // 除外する小計行の「値」の集合（件数ではなく値で判定）
     const SUBTOTAL_NAMES = new Set(['総額', '保険外のみ', '']);
     const isSubtotalRow = name => SUBTOTAL_NAMES.has(String(name || '').trim());

     const col = resolveColumns(rows[1]); // findCol で列名→index（位置固定なし）
     const records = rows
       .slice(2)                          // 3行目以降（メタ行+ヘッダー行の2行だけ確定的にスキップ）
       .map(parseLine)
       .filter(a => !isSubtotalRow(a[col.name])) // ← 値ベース除外。行数が変わっても壊れない
       .map(a => normalize(a, col));
     ```

     > 実データでは小計行がたまたま各月9行だが、**コードは「9」を知らない**。将来 CSV の小計行が増減しても `利用者名` の値だけで判定するため件数非依存。
   - 各行を正規化: `{ tsuki, hken, name, riyou:Number, jihi:(税抜+消費税), pay, nyukin }`。
4. `mergeMonths(files[])` → **`被保険者番号` をキーに縦マージ** → `{ 利用者: { 202603:cell, 202604:cell, … } }`。同一 `(被保険者番号 × 月)` が複数行に出た場合は請求額・自費を**加算**（実データは重複ゼロだが、将来のサービス種類分割等に備え加算ロジックを残す）。
5. `classify(cell)` — **A案（確定・排他条件・5状態）**:

   5状態（`unpaid` 🔴 / `pending` 🟡 / `exempt` ⬜ / `paid` 🟩 / `empty` 空）に**必ず1つだけ**割り当てる。排他性は**評価順**で保証する。

   **入金状況の実値は3系統ある（実データで確認）**:
   - `入金済` → 🟩 paid
   - `未入金` → 🔴 unpaid（引落し失敗 or 請求確定済・引落し待ち。6月がこれ）
   - **`''`（空欄）→ 🟡 pending（請求確定前＝判定不能）**。3月旧ファイル(2026-04-12出力)は個人行全115が空欄。まだ入金/未入金を断定できない「判定前」状態として明示する（`未入金`＝赤 とは別物）。

   境界の転び先を評価順で固定する：
   - **「請求額0」は入金状況が空欄でも `未入金` でも、必ず `exempt`**（`riyou === 0` を最初に判定）。
   - **「空欄 かつ 請求>0」は `pending`**（赤にしない・緑にしない）。

   ```js
   function classify(cell) {
     if (!cell) return 'empty';                 // その月にその人の行が無い
     const riyou  = cell.riyou;                 // 数値（利用者請求額）
     const nyukin = String(cell.nyukin || '').trim(); // '入金済' | '未入金' | ''

     if (riyou === 0) return 'exempt';          // ⬜ 対象外（公費のみ・利用者負担ゼロ）
     //   ↑ 請求0を全部吸収。「請求0×未入金」「請求0×空欄」もここで exempt。
     if (nyukin === '')     return 'pending';   // 🟡 判定前（請求確定前＝空欄・引落し結果まだ）
     if (nyukin === '未入金') return 'unpaid';   // 🔴 未入金（riyou>0 が保証済み）
     if (nyukin === '入金済') return 'paid';     // 🟩 入金済
     return 'unknown';                           // ⚠️ 上記以外の想定外値は握りつぶさず可視化
   }
   ```

   | 入力 | riyou | nyukin | 結果 |
   |---|---|---|---|
   | 通常の未入金 | 2920 | 未入金 | 🔴 `unpaid` |
   | 入金済 | 3197 | 入金済 | 🟩 `paid` |
   | **請求確定前（空欄）** | 4535 | `''` | 🟡 `pending`（判定前・赤にしない） |
   | **境界: 請求0×未入金** | 0 | 未入金 | ⬜ `exempt` |
   | **境界: 請求0×空欄** | 0 | `''` | ⬜ `exempt` |
   | 生活保護（公費のみ） | 0 | 入金済/未入金/空欄 | ⬜ `exempt` |
   | 想定外値 | 5000 | `保留` 等 | ⚠️ `unknown`（データ異常の可視化） |
   | 行なし | — | — | 空 `empty` |

> 実データ裏取り①（令和8年5月・引落し後）: 「未入金」6人のうち請求>0の **川島勝子・町田和子の2人だけが真の赤**。残り4人は請求0円＝`exempt`。
> 実データ裏取り②（令和8年3月・旧2026-04-12出力・請求確定前）: 個人行全115が空欄 → 請求>0の110人が 🟡 `pending`、請求0の5人が `exempt`。**この月を入れても誰も赤にならない**（判定前だから）。これが「まだ判定できない」の正しい可視化。
> ※ `pending` と `unknown` は別物: `pending`＝想定された正常な「判定前」（空欄）、`unknown`＝入金状況が想定外の文字列だったときのデータ異常セーフティネット。

## 5. 表示仕様

- **マトリクス表**: 縦=利用者、横=月（左=古い → 右=新しい、例 3月・4月・5月）。
- **各セル**: 利用者請求額を表示。状態ごとに配色（5状態＋空）:

  | 状態 | 表示 |
  |---|---|
  | 🔴 `unpaid` 未入金 | 赤背景 |
  | 🟡 `pending` 判定前 | 淡い黄/グレー・ラベル「—（判定前）」 |
  | 🟩 `paid` 入金済 | 淡緑/通常 |
  | ⬜ `exempt` 対象外 | 薄グレー |
  | ⚠️ `unknown` 想定外 | 太枠・警告アイコン（データ異常を目立たせる） |
  | 空 `empty` | 空セル |

  自費があれば小さく併記。支払方法アイコン（口座/現金/振込）。
- **ソート**: 「未入金がある人」を上に（社長の主目的）。名前順トグルも用意。`pending` はソートで赤の下（未確定は追跡対象ではない）。
- **サマリ**: ヘッダーに未入金の人数・合計額。任意で月別の請求合計・未入金合計行/列。
- **注意書き（常設1行）**: 「入金状況が**空欄の月＝請求確定前**は『—（判定前）』表示で入金判定しません。『未入金』は引落し失敗 or 引落し待ちを含みます。確定した入金/未入金は**引落し後のCSV**で反映されます。」

## 6. テスト（TDD）

- 純関数（decode / parse / extract / merge / classify）を Node でユニットテスト。実CSV（令和8年3・4・5・6月＋3月旧ファイル）をフィクスチャに既知値でアサート:
  - **令和8年5月（引落し後）**: 個人 116人、🔴未入金 = **2人（川島・町田）**、⬜対象外0円 = 4人、小計除外 = 9行。
  - **令和8年6月（請求確定済・引落し前）**: 個人 111人、入金状況は全員 `未入金` → 請求>0が 🔴 unpaid・請求0が exempt。**pending は0**（空欄ではないため）。
  - **令和8年3月・旧ファイル `_202604121040`（請求確定前・全空欄）**: 個人行全115が空欄 → 🟡 `pending` = **110人**、⬜ `exempt`（請求0×空欄）= 5人、🔴 `unpaid` = **0人**。← 空欄=pending の核心テスト。
  - **classify 単体の境界アサート**: `(riyou=0, nyukin='未入金') → exempt` / `(riyou=0, nyukin='') → exempt` / `(riyou>0, nyukin='') → pending` / `(riyou>0, nyukin='保留') → unknown` を各1件。
  - **横断**: 伊熊二三子など同一人物が複数月に1行で並ぶこと。
  - **被保険者番号キー＋空番号除外**: 小計行（被保険者番号が空）が名寄せに混入しないことを明示的にテスト。
- **テスト期待値は実データ固定値**（116人・2人 等）。⚠️ **CSVを差し替えたら期待値も必ず更新すること**（実データに紐づく固定値のため）。
- 本アプリはPOSTゼロの純表示だが、検証は**純関数テスト**で行い、本番HTMLを実ブラウザでは開かない（memory: 本番htmlを実ブラウザで検証するな）。

## 8. UI注意書き（未入金の意味の一般化）

- 未入金の理由は**多様**（残高不足・口座解約・依頼書不備・死亡 等）。UI文言は「**引落しできなかった人は furikae で追跡／振込等で回収**」と**一般化**し、**「死亡」等の特定理由に限定した文言にはしない**。
- 「引落し不能 → 振込回収」の導線を注意書きで示すが、回収手段・理由を決め打ちしない。

## 9. furikae との住み分け（正式）

調査（2026-07-18・`furikae.html` 実装確認済み）に基づく公式方針。

### 9-1. 役割の線引き

| | **seikyu-board（発見の鏡）** | **furikae（追跡の台帳）** |
|---|---|---|
| 性質 | **read-only** | **書込あり**（localStorage + GAS永続） |
| データ源 | リハブ **請求情報CSV** | 電算 **振替結果Excel**（XLSX） |
| 識別キー | **被保険者番号**（例 `0000100038`） | **電算の顧客番号**（例 `149`・小整数） |
| 見るもの | 全利用者×月の請求・入金/未入金の俯瞰 | 振替不能になった案件の理由・繰越・消込 |
| 時点 | 請求〜引落し前後（俯瞰） | 引落し**後**の不能のみ（案件追跡） |

### 9-2. v1 では両アプリを連携しない

- 二重管理を避けるため、seikyu-board の 🔴（未入金）セルには furikae への **リンク導線のみ**を置く。
- **データは渡さない・画面遷移だけ**（`furikae.html` を開くリンク）。自動登録・自動同期はしない。

### 9-3. キー体系が別＝機械的突合は不可

- **被保険者番号（`0000100038` 形式）≠ 電算の顧客番号（`149` 等の小整数）**。共通項は**氏名のみ**。
- furikae の消込・繰越・冪等処理は**すべて顧客番号キー**（`furikae.html` の customerId = 電算Excel row[7]）。
- seikyu-board は顧客番号を持たないため、**電算取込の代替にはならない**。仮に氏名だけ送っても furikae の消込チェーンには乗らない。

### 9-4. 将来ちゃんと連携する場合の起点（v1スコープ外）

- 前提として **被保険者番号 ↔ 顧客番号のクロスウォーク表を1本用意**すること。
- その上で（a）furikae 側に「1件登録の受け口（doPost）」新設、（b）連携方向は seikyu-board →（既存 furikae GAS・同オリジン）→ furikae 台帳、が必要。
- v1 では着手しない。

## 10. v1 スコープ外（次段以降）

- 手入力（繰越・返金・備考）の永続化（GASなし方針だと CSV 持ち回り or エクスポートで実現・別途設計）。
- **月またぎの一括回収の追跡**（例: 6月未収 → 8月一括回収を1案件で追う）は **furikae の領分**。seikyu-board では扱わない。
- 電算結果CSVを直接読むリアルタイム判定モード（指示書のB案）。
- furikae 連携（§9-4）。
- 利用額/自費の内訳フル表示（保険請求・公費・負担額等の展開）。
