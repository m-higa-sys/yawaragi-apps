# intake 経営ダッシュボード（P5）設計

- 日付: 2026-07-11
- 対象: ボードGAS（新action `intake_dashboard`）／`intake.html`（新タブ「経営」）／`version.txt`
- 前提スキル出力: brainstorming → 本doc → writing-plans
- 関連: [[project_intake案件台帳化-P1完了-P5宿題]] / [[project_週間予定表リビルド-P2]]（空き枠計算流用）/ P2.1 intake auth ゲート

## 1. 目的

社長の経営判断用の数字を `intake.html` 内の新タブ「経営」に表示する。鍵の中（P2.1 adminKeyゲート）なので氏名も出せる。優先順位（上から）：①需給対比 ②所要日数 ③問合せ元別 ④転換率・失注理由。

## 2. スコープと非対象

**やる**：
- ボードGASに新action `intake_dashboard`（P2.1ゲートリストに1行追加＝鍵必須）。1レスポンスで5指標を返す。
- 純ロジック `intake-dashboard-core.js`（Node+GAS両対応）に計算を隔離しTDD。
- `intake.html` に新タブ「経営」を追加して描画。

**やらない（非対象・制約）**：
- シート構造変更禁止（読むだけ・既存列のみ）。
- P2.1ゲートのロジック非改変（`intakeAdminAuthorized_`・鍵管理は触らない。ゲート対象リストに1action足すだけ）。
- P1フェーズ遷移履歴の**書込**ロジック（`appendPhaseHistory_`）非改変。読むだけ。
- 既存 `getIntakeFunnel`（カンバンのファネルが使用中）は**非改変で残す**。`intake_dashboard` は別action新設。

## 3. 実データ前提（実測済 2026-07-11・コード.js:125-174）

- **フェーズenum**（列「フェーズ」・現在状態のみ）：受付 / 見学 / 体験 / 契約準備 / 利用開始準備 / アーカイブ / ドロップ
- **日付列**：問い合わせ日 / 予定日 / 見学日 / 契約日 / 重要事項説明日 / 契約書渡し日 / 契約書受領日 / **本格利用開始日** / ドロップ記録日時 / 作成日時 / 更新日時
- **完了フラグ**：見学完了 / 体験打診結果 / 体験完了 / 重要事項説明済 / 契約書取り交わし済
- **分類軸**：連絡元区分 / 利用意向(あり/保留/なし) / ドロップ理由 / 利用者台帳反映済
- **P1履歴**：列「フェーズ遷移履歴」= JSON `[{from,to,at,by,reason}]`。**P1導入は2026-07-10**so既存案件はほぼ空、7/10以降の遷移のみ記録。→ 履歴は「あれば使う」補助であり、既存案件は日付列ベースの概算が主体になる（approxフラグで明示）。

## 4. アーキテクチャ

```
intake.html「経営」タブ
  └ jsonp(BOARD_URL, {action:'intake_dashboard', adminKey})   // P2.1ゲート対象に追加
       └ GAS intake_dashboard(ss, e)
            ├ 認証: intakeAdminAuthorized_(e) 不一致→{error:'unauthorized',status:401}
            ├ 見学体験新規 全行read → 行→案件obj 正規化（日付はyyyy-MM-dd文字列化・履歴はJSON.parse）
            └ intakeDashboard_(cases, today) を呼ぶ
                 └ intake-dashboard-core.js（純・Node+GASで同一コード）が5指標を計算
       └ フロント描画（空き枠だけは週間予定表と同じ在籍daicho計算をフロントで流用）
```

**方針**：壊れやすい計算（所要日数ハイブリッド・転換率の進行中除外）をNode単体でTDDするため純ロジックを `intake-dashboard-core.js` に隔離。GASの `intake_dashboard` は薄いアダプタ（read＋正規化＋core呼び＋返す）。既存の `weekly-core.js` / `intake-auth-core.js` / `intake-phase-history-core.js` と同型。

## 5. 純ロジック仕様（`intake-dashboard-core.js`・各TDD）

### 5.0 正規化された案件オブジェクト（coreへの入力）

GASアダプタが行→objへ変換して配列で渡す。coreはシートを知らない。各案件：
```
{
  氏名, フェーズ,
  見学完了(bool), 体験完了(bool),
  問い合わせ日, 見学日, 予定日, 契約日, 本格利用開始日, ドロップ記録日時,   // yyyy-MM-dd or '' 
  連絡元区分, 利用意向, ドロップ理由, 利用者台帳反映済(bool),
  履歴: [{from,to,at,by,reason}]   // parse済・無ければ []
}
```

### 5.1 `dashStageBuckets_(cases)` — 需給対比（社長補正反映）

除外：フェーズ∈{ドロップ, アーカイブ} または 利用者台帳反映済=true。
残りを次に振り分けて返す：
```
{
  受付:        フェーズ==受付 の件数（別枠表示）,
  進行中: {                                   // 進行中パイプライン（段階別）
    見学予定: フェーズ==見学 && 見学完了!=true,
    見学済:   フェーズ==見学 && 見学完了==true,
    体験予定: フェーズ==体験 && 体験完了!=true,
    体験済:   フェーズ==体験 && 体験完了==true,
    契約準備: フェーズ==契約準備
  },
  開始待ち:    フェーズ==利用開始準備 の件数        // 契約済・台帳/空き枠に未反映＝ダブルブッキング温床so独立表示
}
```
`進行中M`（対比用合計）= 進行中パイプライン5段階の合計。**開始待ち・受付はMに含めない**（別枠）。

### 5.2 `dashLeadTime_(cases)` — 所要日数（ハイブリッド）

対象：**`本格利用開始日` があり かつ 過去日（today以前）の案件のみ**。フェーズ条件は付けない。
理由（社長補正 2026-07-11）：利用開始準備＝開始待ちは終点日が無い/未来so中央値を汚す。開始待ちは需給の「開始待ち」枠で表現済みso所要日数には入れない。本格利用開始日が未来日（予定）の案件も終点未到達so除外。
各案件：
```
{ 氏名, days:(本格利用開始日 − 問い合わせ日), source:'history'|'approx', 段階別?:{...} }
```
- `days` は問い合わせ日→本格利用開始日の全体日数。
- `source`：フェーズ遷移履歴が段階遷移を含めば `history`、無ければ日付列からの `approx`。
- `段階別`：**historyのとき**のみ、遷移`at`の差分で段階別所要（例 受付→見学, 見学→体験…）を分解。approxは全体日数のみ（段階別なし）。
返り値：`{ 中央値(median days), 件数, cases:[...] }`。件数0のとき中央値=null。

### 5.3 `dashConversion_(cases)` — 転換率（累計・進行中除外）

到達判定 `reachedStage_(case, stage)`：履歴のmax到達を優先。無ければ 現フェーズ順位＋日付列/完了フラグで概算。
- 見学到達：履歴to∈見学以降 or フェーズが見学以降 or 見学日あり or 見学完了==true
- 体験到達：履歴to∈体験以降 or フェーズが体験以降 or 体験完了==true
- 契約到達：履歴to∈契約準備以降 or フェーズが契約準備以降 or 契約日あり or 契約書取り交わし済==true

各遷移（見学到達→体験到達／体験到達→契約到達）で：
```
{ 分母, 分子, 率, 進行中N }
分母 = 直前段階に到達 かつ 次段階の成否が確定（次段階到達 OR ドロップ済）
分子 = 次段階に到達
進行中N = 直前段階に到達 かつ 未確定（次段階未到達 かつ 非ドロップ）  // 分母にも分子にも入れない
```
これで「直近に進行中案件が多いと率が悪く見える」問題を回避。全期間・全案件が母集団（月フィルタなし）。

### 5.4 `dashSources_(cases)` — 問合せ元別

- 連絡元区分ごと：`{ 件数, 利用開始数 }`（利用開始数＝その区分でフェーズ∈{利用開始準備,アーカイブ} or 本格利用開始日あり、累計）。
- 月次推移：問い合わせ日の `yyyy-MM` ごとの件数（件数系so月次）。空/未設定区分は `'未設定'` に寄せる。

### 5.5 `dashLostReasons_(cases)` — 失注理由

ドロップ案件（フェーズ==ドロップ）：
- ドロップ理由ごと `{ 件数 }`（空理由は `'未設定'`）。
- 一覧：`[{ 氏名, 到達段階:(reached maxの日本語名), 到達段階approx:bool, 理由, 日付:ドロップ記録日時 }]`（鍵の中so氏名可）。日付降順。
- `到達段階approx`（社長補正 2026-07-11）：**フェーズ遷移履歴が無い＝P1(7/10)前の推定案件は `true`**（日付列/フラグからの概算so到達段階が推定と分かる）。履歴で到達段階を確定できた案件は `false`。UIは approx行に概算印（例「※推定」）を出す。

### 5.6 エッジ

空配列→全指標ゼロ／中央値null／リスト空。履歴JSONが壊れていてもcore手前（アダプタ）でtry-parse→壊れは`[]`扱い。日付パース不能は当該案件を該当計算からスキップ（所要日数・月次で無視）。

## 6. フロント（`intake.html` 新タブ「経営」）

タブ切替は既存タブUIに追随。起動時 or タブ選択時に `jsonp(BOARD_URL,{action:'intake_dashboard',adminKey})`。adminKey付与・unauthorized時は既存の鍵再入力フロー（`getIntakeAdminKey`）に委譲。

**★配置場所の制約**：ダッシュボードのフロントJS/HTMLは **`intake.html` インラインに置く**。理由＝intake.htmlは版ゲート対象（`?v=` で location.replace）だが `shared.js` は `?v=` 無しで読まれる（intake.html:27）so、shared.js に足すと版bumpで更新されずキャッシュ事故になる。共通関数の再利用が必要でも新規ロジックはインライン側に。

描画順（上から）：
1. **需給対比**：`空き枠N ／ 開始待ち ／ 進行中M（段階別内訳）／ 受付` を一目の対比で。
   - 空き枠N＝週間予定表と同じ在籍daicho計算をフロントで流用：`Σ(18 − 各枠現員)` over am/pm×月〜金（在籍daicho＝riyousha-daichou-api既定）。coreには入れない（フロント計算・GAS改修なし）。
   - **★曜日別ampmパースの流用必須（社長補正 2026-07-11）**：各枠現員の判定は**週間予定表で修正した `slotSet_`/`attendsCell_`（曜日別ampmパース）を流用する**。素朴な `ampm.indexOf('午前')` 方式を新規実装しないこと。理由＝複合ampm「月午前、木午後」(宮さん)の幽霊計上が版47で根治済みso、ダッシュボードで素朴実装すると同じ幽霊が再発する（[[project_週間予定表リビルド-P2]] 参照）。実装＝これらの純関数を共有できる形（intake.htmlインライン内に同ロジックを持つ or 共有元から流用）にし、週間予定表と同一の空き枠数になることをテスト/実機で突合。
2. **所要日数**：中央値＋件数を大きく。下に案件一覧（氏名/days/段階別）。`source=='approx'` 行に「※日付列からの概算」注記。
3. **問合せ元別**：区分ごと 件数＋利用開始数。加えて月次問合せ推移（棒/折れ線）。
4. **転換率**：見学→体験／体験→契約 の累計率。各に「進行中N件」を別枠表示（率の分母外と明記）。
5. **失注理由**：理由別件数の下に、1件ずつのリスト（氏名・段階・理由・日付）。この規模では率より個別が雄弁。

率系は累計を既定、件数系（問合せ数）は月次推移。

## 8. テスト計画（TDD必須）

`scripts/test-intake-dashboard-core.js`（Node・jsdom不要）：
- 需給：フェーズ×完了フラグの全分岐→正しいバケット。ドロップ/アーカイブ/台帳反映済の除外。開始待ち・受付の別枠。
- 所要日数：**履歴あり案件（段階別分解が出る）と日付列のみ案件（approx・全体日数のみ）の両方**。中央値の偶数/奇数件。対象0件→null。**対象＝本格利用開始日が過去日のみ**＝未来日（開始予定）・開始待ち（利用開始準備で終点なし）が中央値に入らないことをテストで実証。
- 転換率：**進行中案件が分母にも分子にも入らないことを実証**。ドロップ案件が「到達段階まで」分母に入ること。全確定時の率。
- 問合せ元：区分集計＋利用開始数＋月次推移。未設定寄せ。
- 失注：理由別件数＋一覧＋日付降順。空理由の未設定寄せ。
- エッジ：空配列・壊れ履歴・日付パース不能。

ランタイム実測（完了条件）：`intake_dashboard` 鍵なし→unauthorized／鍵あり→success（**鍵は社長のみ保有so最終実測は社長依頼**）。実データでタブ表示の件数が実シートと一致することを確認（社長）。
