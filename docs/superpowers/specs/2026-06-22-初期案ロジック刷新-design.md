## 1. 勤務ルールのデータモデル案

スタッフ単位で「固定型」「週N回型」「管理者型」を**1つのスキーマで表現**する。

```js
const WORK_RULES = {
  // ① 曜日固定型（大久保さんなど）
  "大久保": {
    type: "fixed",
    days: { 1:"B2", 4:"B2" }        // 曜日(1=月)→勤務パターン
  },

  // ② 週N回＋曜日希望＋連勤上限型（春山・髙山・石井・下浦など）
  "春山": {
    type: "weekly",
    count: 3,                       // 週の出勤回数
    pattern: "E1",                  // 基本の勤務記号（=勤務時間帯）
    prefDays: [2,4,5],              // 希望曜日（優先順・火木金）
    ngDays: [],                     // 出られない曜日
    maxConsecutive: 2,              // 連勤上限（3連勤NGなら2）
    contract: { noWeekend:true, noHolidayWeekday:false },
    timeRule: { extend:"F", maxExtendPerWeek:2, noConsecutiveExtend:true },
    monthOverride: { "2026-07": { timeRule:{ extend:"F", maxExtendPerWeek:2 } } }
  },
  "髙山": { type:"weekly", count:3, pattern:"E1", prefDays:[1,3,5],
            contract:{ noWeekend:true, noHolidayWeekday:true } },  // 土日祝休み

  // ③ 管理者型（社長）= §1a
  "社長": {
    type: "manager",
    count: 5,                       // 週5
    maxConsecutive: null,           // 連勤上限なし
    roles: ["相談員","機訓","フロア"], // 看護師は不可（資格なし）
    pattern: "B2",                  // 現場に入る時間帯（仮）
    adminReserve: true,             // 管理業務(請求/労務/記録)の時間を確保＝フル現場固定にしない
    floorCountPolicy: "managerSafe" // 現場カウントは「管理業務に支障がない範囲」
  }
};
```

### 1a. 管理者（社長）の扱い
- **週5・連勤上限なし**。役割候補は **相談員・機能訓練指導員・介護職員（フロア）**。**看護師は不可**（看護資格なし）。
- ただし**管理業務（請求・労務・記録等）があるため、フル現場固定にはしない**。配置上は「相談員/機訓/フロアの**充足要員候補**」として扱いつつ、管理業務の時間を確保する前提（現場カウントは管理業務に支障がない範囲＝`floorCountPolicy:"managerSafe"`）。
- 用途イメージ: 相談員1+や機訓2+、フロア5+が**他で埋まらない日に社長を充当**する“最後の埋め合わせ要員”。常時フル投入はしない。

ポイント（全型共通）:
- 役割（看護師/介福/相談員/送迎…）は既存定数（NURSE 等）を流用、または各ルールに `roles:[]` を持たせ一元化。
- 希望休・外せない予定（WISH_DATA）／承認済み要望（conditions）は**別レイヤー**として配置時にハード/ソフト制約として流し込む（月ごとに変わるためルール本体に埋めない）。
- 月固有の上書きは `monthOverride: { "2026-07": {...} }`。

### 1b. 兼務ルール（基準準拠）
- **管理者は、管理上支障がない範囲で、相談員・介護職員・機能訓練指導員を兼務可。**
- **看護職員は提供時間帯ずっと専従である必要はない＝必要時配置でよい**（だから看護師1+を「その日1名いればOK」で運用できる）。
- **相談員・機能訓練指導員は他職種と兼務可。**
- ⚠️ **注記: 最終的な兼務可否・常勤換算（FTE）カウントは東松山市の運営基準・指導に従う。本設計の兼務ルールは「自治体確認の上で調整可」とし、確定値は指導内容に合わせて差し替える。**

## 3. 配置ロジックの方針（チェックを“制約”として使う）

現状の各チェックは「組んだ後に色を付けるだけ＝事後表示」。これを**配置の制約・評価関数に昇格**する。

方針（グリーディ＋制約充足）:
1. 固定型を先に配置（大久保など）。希望休・契約で除外。
2. 週N回型を希望曜日の優先順に仮置きし、各人の `count`・`maxConsecutive`・`ngDays`・希望休を守る。
3. 各日について役割の最低人数を判定（checkAlerts のカウント関数を「評価関数」として再利用）。不足する役割があれば、その役割を満たせる週N回型スタッフの未消化枠を、希望曜日を崩してでもその日へ寄せる（例：月曜に看護師→看護師の週N枠を月曜へ）。
4. それでも埋まらない相談員/機訓/フロアは、**管理者（社長）を `floorCountPolicy:"managerSafe"` の範囲で充当**（最後の埋め合わせ）。看護師の穴は社長では埋められない（資格なし）ので穴のまま明示。
5. 解けたら確定、解けなければ穴として残す（§4）。

実装の肝: **checkAlerts の集計ロジック（role→count）を純粋関数に切り出し、「配置中の評価」と「事後表示」で共用**する。表示と配置の判定基準を必ず一致させ、二重実装を避ける。

## 4. 制約が両立しないときの優先順位・穴の扱い

ハード制約（絶対に破らない）:
- 希望休／外せない予定、契約（髙山＝土日祝休）、運転不可、連勤上限、週N回の上限、**資格（社長は看護不可）**。

ソフト制約（目標。両立しなければ緩める）。緩める順序の提案:
1. 看護師1+（人員基準・最優先で死守。社長では埋められない＝看護師の週N枠/希望曜日を最優先で寄せる）
2. 送迎3+ / フロア5+（運営の安全）
3. 機訓2+ / 介福2+ / 相談1+ / 15時以降4+（サービスの質。社長を管理業務に支障ない範囲で充当）
4. 本人の曜日希望（最後に妥協）

埋まらない日の扱い:
- 自動で人を捏造補充しない。穴は「🔴 7/20(月) 看護師ゼロ」のように明示し `#alertBox` に表示。
- あわせて「誰を入れれば埋まるか」の候補を提示（例：7/20＝春山 or 石井を月曜に追加／相談員不足なら社長を充当）。最終判断は社長。

## 8. Phase 1 詳細設計（weekly型の自動割当）

**目的**: weekly型スタッフを「週N回・希望曜日・連勤上限」に基づいて**自動で曜日に割り付ける**。Phase 0 と違い**出力は変わる**（賢くなる）。役割最低人数の制約化は Phase 2、社長の穴埋め充当も後Phase。本番投入は**7月シフト完成後**。

### 8.1 対象と非対象
**自動割当の対象（type を weekly 化）**:
| スタッフ | count（週何回） | 連勤上限 maxConsecutive | 既定 prefDays（希望曜日） | pattern |
|---|---|---|---|---|
| 春山 | 3 | 2（3連勤NG） | 火木金 | E1 |
| 髙山 | 3 | 2（3連勤NG） | 月水金（土日祝NG=契約） | E1 |
| 石井 | 3（2〜3・基準3） | 2（3連勤NG） | 火木 | E1 |
| 下浦 | 3 | 2（3連勤NG） | 火水金 | H |
| 工藤 | 3（2〜3・基準3） | なし（回数の週3上限のみ） | 月水木 | H |
| 小野 | 4 | なし | 月〜金から4日 | 7① |
| 林 | 2 | なし | （個別実績→既定prefDaysに転用） | 7① |

**非対象（Phase 1 でも据え置き）**:
- 週5社員（勝又・星野・比嘉）＝ fixed の月〜金（週5＝全営業日。自動割当不要）。
- 大久保 ＝ fixed（唯一の真の固定）。
- 社長 ＝ manager（**定義のみ・自動配置しない**。穴埋め充当は Phase 2 以降）。

※ 林は現 individual（INDIVIDUAL_DAYS）。Phase 1 で weekly(count2) に移行し、既存の林の INDIVIDUAL_DAYS は廃止 or 既定 prefDays の初期値に転用する。

### 8.2 入力データ
- WORK_RULES: type / count / pattern / prefDays / ngDays / maxConsecutive / contract。
- 希望曜日(prefDays): 毎月シフト希望入力アプリから取得（§8.5）。取得不可/未入力の月は WORK_RULES の既定 prefDays。
- 希望休・外せない予定: WISH_DATA[ym]（既存 getWishes）。
- 営業日: 既存 buildCalendar（isOpen = dow !== 0 ＝日曜休館）＋ HOLIDAYS。

### 8.3 週の定義（日曜始まり）
- **週境界＝日曜始まり（日〜土）**。JS getDay() 0=日 と整合。
- 月内を日曜始まりの週に分割。月初・月末は部分週になる。
- 各週の「対象営業日」＝その週の isOpen 日（日曜除く＝月〜土の開館日。祝日も開館）。
- 週N回(count)は**フル週(営業6日)基準**。部分週は按分: weekCount = round(count * 対象営業日数 / 6)（0〜対象営業日数でクランプ）。按分式は社長確認で調整可。

### 8.4 割当アルゴリズム（グリーディ＋連勤回避）
処理順: ①fixed/週5社員/大久保を先に確定 → ②weekly型を割当（Phase 1 は役割制約なしのためスタッフ順は固定でよい）。

各 weekly スタッフ × 各週:
1. candidate = その週の対象営業日 − 希望休 − 外せない予定 − ngDays − 契約NG（髙山は祝日も除外）。
2. prefDays 該当日を優先順に並べ、続けて残りの営業日を後段に並べる。
3. weekCount 個を選ぶ。選択時に**連勤上限**を満たすよう調整:
   - 候補を順に仮置きし、その日を入れると連続出勤が maxConsecutive を超える場合はスキップして次候補へ。
   - 希望曜日が連続して上限超過するときは間引いて非連続化（例: maxConsecutive=2 で月火水希望なら月火＋別日 等）。
4. 選んだ日に pattern を割当。
5. weekCount に届かない場合（希望休過多等で候補不足）＝その週は**不足のまま（穴）**。ログ＆警告に残す（捏造補充しない）。
6. 連勤は日曜休館で自然にリセット（土→翌週月は日曜を挟むので非連勤）。

### 8.5 曜日希望の取得（毎月・シフト希望入力アプリ）
- 現状 getWishes は希望休(wishes)・外せない予定(absences)のみ。**「出勤希望曜日」は未収集**。
- **前提整備（別作業）**: シフト希望入力アプリ(shift.html)に「希望出勤曜日」入力UI＋GAS API（getPrefDays 新設 or getWishes 拡張で prefDays:{staff:[dow...]} を返す）を追加。
- 取り込み: 月初期化時に prefDays を取得し WORK_RULES の prefDays を上書き。取得失敗/未入力は既定 prefDays。曜日は日曜=0基準（§8.3と整合）。
- **段階化**: まず既定 prefDays で自動割当を動かす **Phase 1a** → アプリ取得連動を **Phase 1b**（前提整備後）。

### 8.6 検証計画（制約充足・回帰ではない）
出力が変わるため一致比較は不可。新出力が各制約を満たすかを検証する scripts/test-workrules-phase1.js（2026-05/06/07で実行）:
- **C1 週N回**: 各 weekly スタッフ × 各週の出勤回数 == 按分 weekCount（フル週=count）。
- **C2 連勤上限**: maxConsecutive 該当者（石井/下浦/春山/髙山）は連続出勤 ≤ 2。違反 0。
- **C3 希望休尊重**: 希望休・外せない予定の日に出勤 0。
- **C4 契約**: 髙山が土日祝に出勤 0。
- **C5 ngDays**: 禁止曜日に出勤 0。
- **C6 非対象の不変**: 週5社員・大久保の fixed 出力が Phase 0 と一致（部分回帰）。社長は配置 0（manager 未配置）。
- **C7 希望曜日カバレッジ（参考・合否外）**: 選択日のうち prefDays に合致した割合を出力（品質可視化）。
- 全 assert（C1〜C6）を満たせば PASS。C7 は参考値。

### 8.7 役割充足の扱い（Phase 1 範囲外）
- 看護師1+/フロア5+ 等の役割最低人数は Phase 1 では**従来どおり事後警告（checkAlerts）のまま**。配置制約への昇格は Phase 2。
- よって 7/20 のような穴は Phase 1 でも穴のまま（警告表示）。自動回避は Phase 2。

### 8.8 変更範囲・リスク・ロールバック
- **変更**: WORK_RULES（対象を weekly 化）＋ initShiftData の曜日展開部（rulePatternForDay を「週割当の結果参照」に拡張、または initShiftData 内に週割当ループを新設）。

- **リスク**: 中（割当結果が変わる）。本番投入は7月シフト完成後・社長の実機確認必須。
