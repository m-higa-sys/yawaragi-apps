- 作成日: 2026-07-09
- 位置づけ: **設計書作成のみ**。実装・writing-plans（実装計画）には進まない。社長レビュー待ちで停止する。
- 前提: 会話で合意済みの確定内容を構造化したもの。新規の設計判断はしていない。
- 本番非接触: 本番アラート `getUsageAlerts` / `judgeUsageBadgeV2` には**一切触れない**。

---

## 1. 目的

利用者の利用頻度を数字で把握し、契約回数の見直し（**減らす / 増やす**）と曜日別の弱点を、
感覚でなく数値で判断する。**両方向（下げる / 上げる）のダイヤル**として機能させる。

最終的な位置づけは「相談テーブルに載せる候補」を出す指標であり、最終判断は本人希望・ケアマネ込み。

---

## 2. アーキテクチャ（3層分離）

本番アラート（`getUsageAlerts` / `judgeUsageBadgeV2`）とは**完全分離**。既存本番ロジックは無改修。

### 2-1. dump層（GAS・読み取り専用）

- 関数: `dumpUsageInputs(fromYM, toYM, asOf)`
- 役割: 利用者 × 期間の**生入力だけ**を JSON 出力する。判定も率も計算しない。**書込ゼロ**。
- 併せて出すもの: **「理由語彙一覧」（distinct reason ＋ 件数）**。
  - 目的 = 欠席理由 → 除外/カウント の対応表を**実データで確定**するため。

### 2-2. 純関数層（Node・テスト対象）

- ファイル: `usage-frequency.js`
- 役割: dump JSON **だけ**を入力に v1.1 全定義を計算する。
- `SpreadsheetApp` 非依存 ＝ ローカルテスト可（認証不要・本番非接触）。

### 2-3. 算出ドライバ（Node）

- 役割: 全員を表計算し、**実質週回数 / 率 / n / 除外件数 / 判定 / 方向 / 曜日別**を出力。
- 既知ケース突合 ＋ 旧 V2 差分もここで行う。

---

## 3. 指標定義

| 指標 | 定義 |
|---|---|
| 全体利用率 | 出席回数（**追加利用含む**） ÷ 予定回数 |
| 実質週回数 | 契約週回数 × 全体利用率 ← **表示の主役**（例 0.8回/週） |
| 曜日別利用率 | 各曜日の出席 ÷ 各曜日の予定 |

> **罠回避**: 実質週回数は必ず「契約週回数 × 率」で出す。`÷4.3` 等の除数を発明しない。
> （率 ＝ 出席 ÷ 予定 なので、週数は約分で消える。除数を勝手に入れると二重割りになる）

---

## 4. 分母（予定回数）

```
予定回数 = 契約曜日の該当日数（窓内） − 除外日
```

- 除外日: **休業日 / 施設側中止 / 入院・長期不在の該当日**
- **除外日は「分母」と「来られた機会」の両方から引く**。
  - 分母だけ引くとズレる（機会側を引かないと出席率が実態とずれる）。

---

## 5. 追加利用（＝契約曜日以外の来所）※実データ確認済み・確定

- **取得可能**。ソース = **dailyOps**（出勤送迎表GAS `?action=getOps` / SS `1-CryIbG…` シート「出勤送迎表データ」A1）。
- dailyOps は**過去日を累積保持**する（`saveToNisshiSheet` は日付キーで merge-only・間引きなし。overrides の `_pruneSchedOverrides` のような過去日削除は dailyOps には無い）。
- `getMonthlyUsage()`（`gas/yawaragi-board/コード.js`）が **契約曜日 × 欠席 × dailyOps実来館**の突合を既に実装済み。
  - 13780 行「利用予定外の日に来館（イレギュラー出席）→ days に追加」で予定外来所を検出している。
- → **dump は「来所実績日」フィールドを持つ**。これにより率 >100% が成立し、**増やし判定が生きる**。

> 取れないソース（対照）:
> - **overrides**: `_pruneSchedOverrides` で未来日 only に間引かれる（過去3ヶ月は遡れない）。
> - **出欠変更シート**: type 語彙は `欠席` / `長期休み` のみ。追加利用/来所/臨時/出席は記録されない。

---

## 6. 欠席理由の扱い

対応表（テーブル）は**設定で持つ**。区分追加時は**表だけ直す**（ロジックを触らない）。

| 区分 | 理由 |
|---|---|
| 除外 | 入院 / 施設側中止 / 長期不在（ショート等） |
| カウント | 体調不良 / 本人都合 / 家族都合 / 通院 |

- **体調不良はカウント**する。「来れないなら減らす」の対象そのものなので除外しない。
- ※「本人・家族都合」は対応表では **`本人都合` / `家族都合` の2トークンに分割**して持つ（実欠席データが「本人・家族都合」1語で入っていた場合はこの表で未分類に落ちるため、G1/G2 clasp窓の理由語彙一覧で実在文字列を確定する）。
- **未知理由（未分類）の安全弁 ＝ 実装確定（v1で持つ・保留）**: 理由語彙一覧で**全マップするのが本筋**だが、万一の未分類は次の二段で守る:
  1. **黙って率を下げない**: 未分類の欠席日は**分母から除く**（除外と同じ数学的扱い）。ただし除外(入院等)とは別カウント `unclassifiedCount` で数える。＝未知の理由で率が勝手に下がらない。
  2. **減らし判定を保留**: 窓内に未分類が1件でもあれば judge は率判定より前に **`保留`**（第6の判定値）を返し、**減らし/適正/増やし の自動判定に落とさない**。＝誤って減らし候補に落とさず、人が確認する側に回す。
  - 未分類ゼロなら発火せず無害（＝通常運用では出ない安全弁）。保留は閾値でなく原理なので実データ不要。

---

## 7. n（母数）閾値

- 除外後の予定回数 `n === 0`（測れる機会ゼロ・率算出不能） → 「**対象外**」
- `1 ≤ n < 6` → 判定せず「**データ不足**」（＝この字面は n=0 を含まない。n=0 は上の対象外が優先）
- 全窓が閾値割れ（入院等・全窓が対象外/データ不足） → その利用者は総合「**対象外**」（窓横断はドライバ層で集約）

---

## 8. 判定（現契約中心・対称・両方向）

**評価順序**（judge の返す判定値は6種: 対象外 / データ不足 / 保留 / 減らし / 適正 / 増やし）:
1. `n === 0`（率算出不能） → **対象外**
2. `n < 6`（n下限） → **データ不足**
3. **未分類件数 > 0** → **保留**（率判定より先に。§6 の安全弁。減らし/適正/増やしに自動で落とさない）
4. 率 < 70% → **減らし候補**
5. 率 > 110% → **増やし候補**（§8-1 両立案）
6. 適正圏(70〜110)だが 保険(低契約×率天井継続) → **増やし候補**
7. それ以外 → **適正**

「対象外 / データ不足 / 保留」を**率判定(4〜7)より先に**評価する。

| 率 | 判定 |
|---|---|
| 率 < 70% | **減らし候補**（実質週回数を併記。例 実質 週1） |
| 70〜110% | 適正 |
| 増やし候補 | 下記「両立案」 |

### 8-1. 増やし候補の両立案（片張りしない）

- **主**: dailyOps 実利用率で **率 >110%**
- **保険**: dailyOps カバレッジが薄い/欠ける人は「**低契約 × 率が天井≥90%継続**」を併用
  - 理由 = dailyOps に予定外来所が載らない運用だと**過小計上**になるため、片張りにしない。

### 8-2. 方向マーク

- 直近1ヶ月 > 3ヶ月平均 → **↑**（回復傾向）
- 直近1ヶ月 < 3ヶ月平均 → **↓**
- 等しい（または比較不能=null） → **マーク無し（空文字 `''`）**

### 8-3. 曜日別

- ある曜日が全体 **−20pt 以上** かつ その曜日 **n ≥ 4** → 「**○曜 要調査**」

### 8-4. 判定の罠回避

- **丸めは表示の瞬間だけ**。途中で丸めると 70/110 をまたぐ人で判定が反転する。
- **ゼロ除算（n=0）は NaN / Infinity / 0% を返さず、必ず「対象外」に落とす**。

---

## 9. 表示（3列 ＝ 直近1 / 2 / 3ヶ月・トレイリング ＝ 広げた平均）

- 各セル主役: **実質週回数**（小数第1位）例 `0.8回/週`
- 副表示: 全体利用率% ／ 母数 n ／ 除外件数（例 `48% ・n=25 ・入院除外3`）
- 色:
  - 適正 70-110 → 通常
  - 減らし <70
  - 増やし >110
  - 対象外 → グレー
- 判定列: バッジ ＋ 方向マーク（↑ / ↓）

> 位置づけ再掲: これは「相談テーブルに載せる候補」を出す指標。最終判断は本人希望・ケアマネ込み。

## 10. 純関数の分解

```
classifyReason(type, reason, table) → '除外' | 'カウント' | '未分類'
  - type === '長期休み' → 除外
  - type === '欠席'     → 対応表で引く
  - 未知               → 未分類（保留）

calcWindow(inputs, 窓, asOf) → { n, attended, rate, excludedCount, unclassifiedCount, byWeekday }
  - 除外日(入院等)は「分母」と「機会」の両方から引く
  - 未分類日(§6)も分母から引く（黙って率を下げない）＝除外と同数学だが unclassifiedCount で別カウント
  - 境界（以上 / 未満）を1回決めて3窓共通にする（ダブりカウント防止）
  - rate = n===0 ? null : attended/n（NaN/Infinity/0% を返さない）

calcActualPerWeek(契約週回数, 率) = 契約週回数 × 率   ※率は分数・÷4.3等の除数を発明しない

judge(率, n, opts) → 対象外 / データ不足 / 保留 / 減らし / 適正 / 増やし
  opts = { 追加利用, 低契約, 率天井継続, 未分類件数 }
  評価順: 対象外(n=0/null/非有限) → データ不足(n<6) → 保留(未分類件数>0) →
          減らし(<70) → 増やし(>110) → 保険(適正圏かつ低契約×率天井継続→増やし) → 適正
  ※追加利用は情報用（率が既に織り込むためメイン判定に効かせない）

direction(直近1ヶ月, 3ヶ月平均) → ↑ / ↓ / ''（等号・比較不能）

worstDayInvestigate(曜日別, 全体, n) → −20pt以上 かつ n≥4 → 「○曜 要調査」
```

しきい値は**1オブジェクトに集約**:

```js
{ 減らし: 70, 増やし: 110, 曜日差: 20, n下限: 6, 期間: 3 }
```

## 11. v1 で割り切る範囲（数字を見てから本実装で戻す）

1. **契約変更**: 窓分割はやらず、変更者は「参考値（除外フラグ）」で表示（影響が局所のため）。
2. **施設側中止**: dump で該当理由が実在しなければ 0 件。実在すれば対応表に「除外」で載せる。
3. **休業日**:
   - 土日は契約外で自然除外。
   - ★**祝日は要確認**＝ yawaragi が祝日休業か確認し、休業なら祝日を除外日に含める。
   - 直近3ヶ月（4-6月）は **4/29・5/4・5/5・5/6** が窓に入るため、未処理だと率を押し下げる。
   - **年末年始回避だけでは足りない**。

## 12. 検証（数字で示す・自己申告で完了としない）

### 12-1. 罠テスト（合成フィクスチャ・期待値付き）

- `n<6` → データ不足
- 全除外 → 対象外
- ゼロ除算 → NaN / Infinity を出さず対象外
- 入院除外で率が下がらない
- `÷week` 罠（除数を発明していないこと）

### 12-2. 実データ突合

- **週1相当 / 入院 / 回復中（増やし）** の既知3ケースを**名指し**で算出値提示。
- 旧 V2（`judgeUsageBadgeV2`）の出力と**差分表**。
- ※**増やし/回復ケースを必ず1人入れる**（追加利用フィールドが効いている証明になる）。
