## 0. 目的

「今月、誰の・どの書類を送ればよいか」を **利用者 × 書類 × 対象月（何月分）** で一覧化する送付チェックリスト（システムの"背骨"）。3書類（通所／個別機能訓練／口腔）を1リストに集約。

- 状態: `⚠未作成` → `揃った（送付待ち）` → `送付済`
- 「作るべき人」は対象月から**自動**で炙り出す／手動チェックは「揃った」「送付済」の**2つだけ**
- 休みで遅れた分は「○月分・未送付」のまま**翌月以降も繰り越し表示**（＝終わるまで方式）
- 役割: 各担当（通所計画／口腔／個訓）が「揃った」を1タップ → 社長が送信→「送付済」。**最終的には送信も各スタッフへ委譲**。

## 2. 核心の設計判断 — 状態を `(userId, docType, 対象月)` で持つ

**今の問題の根**: 対象月の札がデータに無く、状態が「当月セル位置」に紐づくため繰り越せない。

**解**: 送付状態を **(userId, docType, 対象月)** をキーに明示保存する**追加レイヤ（提出送付台帳）**を新設する。
- 対象月そのものは引き続き算出（満了日 / cycle-judge）でよい。
- "その対象月分を揃えた／送った" の**状態だけ**を対象月キーで持つ。

これで:
```
今月送るリスト = { 行 | 対象月 ≤ 今月 ∧ 状態 ≠ 送付済 }
```
遅延分は対象月ラベルを保ったまま自動で繰り越る。`submitCellColor` の当月限定の弱点を、台帳側が吸収する。

### 提出送付台帳（新シート案・純粋追加）
列イメージ（@279確認後に確定）:
`userId | docType | 対象月(YYYY-MM) | status(未作成/揃った/送付済) | 揃った_at | 送付済_at | 送付者 | 送付方法`
- `送付者` 列を最初から持つ＝**将来のスタッフ委譲**（誰が送ったか）に備える権限/担当の土台。
- `送付方法` は `2026-06-27-soufu-houhoubetsu-group-design.md` の持参→ケアプー→メール→FAX 順に接続。

## 3. 対象月変換層（各書類→台帳の対象月を立てる）

| 書類 | 台帳への対象月の立て方 |
|---|---|
| 通所 評価・計画書 | `getTsushoDueDates` の満了日(YYYY-MM-DD)→ 月(YYYY-MM)。満了日が無い人は暫定として `monitoringFinalEvalMonth` フォールバック（handoff Task A の発展） |
| 通所 モニタリング | 当月（要支援のみ） |
| 個訓 計画 | `isPlanMonth` が真の年月を当月分として stamp |
| 個訓 評価 | `isHyoukaMonth` が真の年月を当月分として stamp |
| 口腔 | `isOralEvalMonth` が真の年月を当月分として stamp |

> サイクル系（個訓・口腔）は純関数で「過去月が対象月だったか」も再計算できるので、台帳に状態が無い過去月分も「未作成」として遡って出せる。出す遡及の地平（何ヶ月前まで）は運用で決める（例: 直近3ヶ月）。
>
> **独立検算での補正3点**:
> 1. 通所評価月の3段階優先: **手動override ＞ 実満了日 ＞ planStart+11計算**
> 2. 遡及地平は書類種別で分ける: **満了日ベースの書類は完了まで永続**、サイクルベースは地平で切ってよい
> 3. **台帳の凍結レコードと、未作成の生再計算を分離**する（後からplanStartが変わっても、確定済み台帳レコードが壊れない構造にする）

## 4. 状態モデル（3状態）＋役割

- `⚠未作成`（自動）: 対象月該当 ∧ 台帳に「揃った/送付済」記録なし。作る側は自動で炙り出す＝**作り忘れの穴を開けない**。
- `揃った（送付待ち）`（担当が1タップ）: 各書類の担当が自分のアプリで押す → 台帳 upsert。
- `送付済`（社長が1タップ・当面）: 送信後に押す → `送付者` 記録。将来は各スタッフが押せるよう権限出し分け。

色は既存 `submitCellColor` を踏襲し、`揃った=green / 送付済=blue / 未作成=red` を維持しつつ「揃った」を中間に正式化。
