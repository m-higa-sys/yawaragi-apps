# 横断ビュー（④ケアマネ送付チェックリスト 格上げ）設計書

- 作成: 2026-06-25
- 対象: `ケアマネ送付チェックリスト.html`（フロントのみ）
- 種別: 既存④の格上げ（新規アプリではない）

## 1. 目的

実績作業のとき「**今月どの事業所に・誰の・何の書類を出すか／出したか**」を1画面で見る、読み取り専用の横断ビュー。これが揃うと「漏れゼロ」が感覚でなく画面で確定する。

## 2. 確定スコープ（ガードレール）

- **④のフロント（HTML/JS）のみ改修。本番GASは一切書き換えない**。既存の get 系を読み取りで叩くだけ。
- 検証は実測ベース。「できました」の自己申告禁止 ── 画面とネットワークタブで確認。
- 本番GASへの push は発生しない（GAS無変更）。④HTML／shared.js の配信は社長承認のうえ別手順。

## 3. 現状（実測で判明）— ④は大半が実装済み

精読の結果、社長指示の機能の**多くがすでに④に実装済み**だった：

| 指示項目 | ④の現状 | 出典 |
|---|---|---|
| ①②③データ取得 | getKeikakushoYear / getTsushoPlans / getOralPlans / contacts 取得済 | fetchAll() 周辺 |
| 当月判定（フロント） | isHyoukaMonth / isMeasureMonth / `planStart===ym` / ?ym系 で実装済 | 判定関数群 |
| 名寄せ | `normOffice`（NFKC＋空白除去）＋送付用居宅一覧の表記へ統一 | canonByNorm |
| 事業所別グルーピング | 実装済（人数順・同一敷地ペア隣接） | grouped[t.office] |
| 赤黄緑の状態色 | 実装済（赤=区変/未・黄=保留・緑=送付可） | user-row-* 帯 |
| 未提出フィルタ | 実装済（実績OK者だけ／全員OKの事業所だけ） | filterFinalizableOnly 他 |
| localStorage送付状態 | `yawaragi_cm_send_progress_v1` 実装済 | STORAGE_KEY |

**明確な未実装は「事業所別⇄利用者別の切替トグル」のみ。** 加えて③通所モニタリング（getMonitoringYear）が未取得。

## 4. データ源（実測済みエンドポイント・すべて本番yawaragiボードGAS / get のみ）

| action | 返す素データ | 事業所キー |
|---|---|---|
| `getKeikakushoYear?year=` | users(planStart, planMonths) / records(keikaku_date, hyouka_*, keikaku_sent_date, tasseido_date) | `cmOffice` |
| `getOralPlansYear?year=` | users(startedAt, evalAnchor) ※records僅少＝口腔計画書運用前。空でも壊さない | `cmOffice` |
| `getMonitoringYear?year=` | users(planStart, finalEvalMonth) / records(pdfSendDate, printSendDate) | **`caremaneOffice`**（他と違う） |
| `getTsushoPlansYear?year=` | users(cycleMonths=12) | `cmOffice` |
| `getTsushoPlans?ym=` | 当月の通所計画対象＋送付状態(planDate, daysSinceCreated) | `cmOffice` |
| `getOralPlans?ym=` | 当月の口腔計画対象＋送付状態(sentToCm, planDate) | `cmOffice` |

| 担当ケアマネ／送付方法 | 利用者台帳「ケアマネ事業所」列＋「送付用居宅一覧」シート（メール/FAX等） | — |

## 5. 当月判定（フロント再判定・新規GAS不要）

| 書類 | 判定 |
|---|---|
| 実績 | 全員・毎月。状態は localStorage から |
| 個訓 計画/評価 | `planStart`/`planMonths` 起点の3ヶ月サイクル（`isPlanMonth`/`isHyoukaMonth`） |
| 口腔 計画 | `getOralPlans?ym=` の当月対象を**優先利用**（フォールバックで `startedAt`＋`isOralEvalMonth`） |
| 通所 計画 | `getTsushoPlans?ym=` の当月対象を**優先利用**。周期は1年固定（近似） |
| 通所 モニタリング | 要介護以外（要支援1/2・事業対象）のみ・毎月。`finalEvalMonth` の月はサイン要（✍） |

`*Year` 系は個訓・通所モニの当月判定に使う。`?ym=` 系は当月対象を直接返すので口腔計画・通所計画はそちらを優先。

## 6. shared.js 1本化の戦略（④先行・①②は次フェーズ）

drift の実害は「**④が個訓の当月判定を正しくやる**」こと。正準ロジック（`isPlanMonth`/`isHyoukaMonth` の planMonths 対応版）は**①が既に持っている**。④の独自 `isHyoukaMonth(planStart, ym)` は planMonths 非対応＝drift。

- **正準の判定関数を shared.js に追加**：`isPlanMonth` / `isHyoukaMonth`（planMonths対応）/ `isOralEvalMonth`。①の実装を移植し、`scripts/test-*.js` でTDD。
- **今回は④だけが shared.js を読む**。④のトップレベル const は shared.js と**衝突しないことを実測確認済み**（`YAWARAGI_BOARD_API` ≠ `YAWARAGIBOARD_API_URL` 等、被りなし）。
- **既存アプリへの無影響は版ゲートで担保**：shared.js は genba 等の本番アプリも読むが、各アプリは `shared.js?v=…` で**版を固定**している。shared.js に関数を**追加**しても、既存アプリが版を上げない限り古いキャッシュを読み続け、挙動は不変。④だけ新版を指して新関数入りを読む。
- **①個訓・②口腔の shared.js 化は次フェーズ**（GAS書き込みタスクの関所と同時に実施。本番運用アプリ2つの配信・回帰を伴うため、今回スコープ外）。

## 7. 名寄せ（必須）

事業所キーがアプリ間で割れている：③（getMonitoringYear）のみ `caremaneOffice`、他は `cmOffice`。横断側で読込時に1キー（`office`）へ正規化してからマージ。利用者の事業所紐付けは**利用者台帳「ケアマネ事業所」列を一次ソース**とし、各APIの office 値と突合（既存 `normOffice`＋`canonByNorm` を踏襲）。

## 8. 表示仕様

- **件数サマリー（常設・実用機能）**：ビュー上部に当月の提出予定件数を常設表示する。`今月の提出予定：計画◯人／評価◯人／実績◯人／通所モニ◯人`。月切替で更新、ビュー切替（事業所別⇄利用者別）では同じ `targets` から数えるので不変。件数定義：
  - **計画** = 個訓の計画月該当（kaigo・`hasNewPlan`。①の `isPlanMonth` と同源）
  - **評価** = 個訓の評価月該当（kaigo・`needsHyouka`。①の `isHyoukaMonth` と同源）
  - **実績** = 当月の対象者総数（全員毎月）
  - **通所モニ** = 要支援/事業対象（`care==='shien'`）の人数
  - このサマリーが①との件数突合（完了条件2）の照合先になる（手計算を廃止）。
- **グルーピング**：送付先ケアマネ事業所ごと（既定）。事業所カードに 事業所名＋提出方法（メール/FAX/持参/ケアプー）＋未提出件数。
- **行**：利用者ごとに、今月の対象書類をチップ表示。
- **状態色**：赤＝未提出/未作成、黄＝遅延（作成済・未送付）、緑＝済。
- **切替トグル（新規）**：事業所別 ⇄ 利用者別（同じデータの並べ替えのみ・**再取得しない**）。
- **フィルタ**：未提出だけ表示（ON/OFF）。
- **既定の月**：システム日付の当月（YYYY-MM）。前月・翌月切替は任意。
- ③通所は1年固定の近似 → 画面下部に「通所は近似（rebuild後に正確化）」の注記。
- 口腔 records が空でもエラーにせず「対象なし」で通す。

## 9. 既知の制約（今回は踏襲・スコープ外）

実績の送付状態は localStorage 由来＝**端末ローカル**。複数端末（Mac/Win）で状態が共有されない。これは既存④の挙動。今回はそのまま踏襲し、GAS/Drive 同期は別タスク。

## 10. 完了条件（実測で確認）

1. 当月、上記エンドポイントから取得→名寄せ→事業所別に集約された画面が出る。
2. ④上部の常設サマリーの「計画◯人／評価◯人」が、個訓アプリ①の当月該当者数（今月計画対象／今月評価対象）と**一致**する（手計算せずサマリー同士を照合）。
3. 事業所キーが正規化され、③（caremaneOffice）の利用者も同じ事業所カードに入る。
4. 事業所⇄利用者 切替、未提出フィルタが効く。
5. ネットワークタブで get 以外（update*/set*/delete*）が**一切飛んでいない**こと。
6. ③通所モニタリングが**「近似表示」の注記として画面に出ている**こと（1年固定の近似である旨を利用者が認識できる）。

## 11. やらないこと（今回のスコープ外）

- GASの新規エンドポイント追加（フロント再判定で足りる）。
- ケアプラン有効期限の保存／`min(ケアプラン, 1年)` 化（通所 rebuild 時）。
- 口腔の実施記録／計画書の分割（別タスク）。
- ①個訓・②口腔の shared.js 化（GAS書き込みタスク直前の関所で実施）。
- リポジトリ stale 修正・clasp 化（同上の関所）。
