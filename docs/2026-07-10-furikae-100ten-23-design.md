## 0. このアプリの本当の仕事（設計思想）

回収は**翌月合算の自動振替**で回る（社長確認済み。現金回収は「振替中止後の最終回のみ」の極めて稀な例外）。
よってこのアプリの仕事は回収督促ではない：

1. **説明責任の管理** — 「翌月2ヶ月分引き落とされる」を利用者に伝え忘れない
2. **翌月成功の見届け** — 合算振替が実際に落ちたかをデータで確認する

伝達ボードは使わない。**スタッフはアプリ（furikae.html）を見れば全て把握できる**形に集約する。

## 1. レイヤー分離（実装単位）

- **②取込の会話導線** = クロコ側（PC上）。「DLした」→ xls発見 → プレビュー → GO → 取込 → サマリ提示
- **③カードの運用機能** = furikae.html + GASシート。やること表示 → 連絡済み記録 → 消込
- 接点は「取込がカードを作る」の一点のみ。別々に実装・テストする。

## 2. ②「DLした」の受け口（クロコ台本）

**原則: ファイル名を信じない。中身を信じる。**

1. `C:\Users\mh\Downloads` を走査。`.xls` のうち**中身が振替結果フォーマットに合致**するものを候補化（列構成・電算顧客番号のゼロ埋め12桁形式で判定）
2. 複数候補は更新日時の新しい順に提示 → 社長が選択（通常は最新1つ）
3. **対象月はファイル内データから読む**（ファイル名からではない）
4. 取込前プレビュー必須: 「対象月◯月・総◯行・うち不能◯件。取り込む?」→ 社長GO → 実行
5. 取込成功後、xlsは `Downloads\処理済み\` へ移動（ログに残す）
6. 複数月の溜まりは既存の冪等設計（対象月＋顧客番号）が守る。二重投入は無害。

**サマリ提示（チャット=レシート、カード=台帳。両方やる）:**

取込完了（◯月分・不能3件・自動消込◯件）
・Aさん ¥1,200 残高不足 → 翌月2ヶ月分の旨を伝える
・Bさん ¥3,400 口座停止 → 口座状態と翌月可否を確認
・Cさん ¥800 理由不明 → 電話で確認
トラッカーにカード3枚作成済み。

## 3. ③カードのライフサイクル（確定）

取込（不能検出）→ カード生成: 未対応
  ↓ スタッフ1タップ
連絡済み（✓日付 スタッフ名）— カードは残る
  ↓ 翌月xls取込時に自動判定（3分岐）
翌月の成功行(code=0)に出現 → 自動消込（人の操作なし）
翌月も不能行に出現         → 「2回目」昇格・金額を累積額に更新・強調表示
翌月どちらにも不在         → ⚠️翌月請求なし・要確認バッジで残す（自動で消さない）

- **消込の主役は人ではなく翌月の取込**。自動消込＝**翌月の成功行(code=0)に顧客番号が出現＝合算成功**（積極シグナル・2026-07-10調査1で4ヶ月実データ100%裏付け・既存 `fnkGoushanCandidates` と同ロジック）。カードの終わり方は3通り：
  - **翌月成功行に出現 → 自動消込**（取込サマリに「自動消込◯件」明示）
  - **翌月も不能行に出現 → 「2回目」昇格・金額を累積額に更新・強調表示**（実測：不能継続は顧客番号あたり1行に累積合算される）
  - **翌月どちらにも不在 → 自動で消さない**。「⚠️翌月請求なし・要確認」バッジで残す（契約終了・現金回収パターンの入口。手動消込の出番）。※「不在」だけを成功と断定してはならない（翌月ファイルに全く現れない＝請求対象外のケースと区別できないため）。
- スタッフの操作は「連絡済み」1タップのみ。消込タップ漏れの運用リスク自体を消す。
- **手動消込ボタンを1つだけ残す**（「現金で回収」「その他理由でクローズ」）。稀な例外の逃げ道。通常フローには出さない・UIを複雑にしない。
- 自動消込は取込サマリに「自動消込◯件（誰）」を明示し社長が目視できる形にする（断定リスクの担保）。

### Step2 実装設計（2026-07-10・「人ごと1枚＋月別内訳」モデル）

**背景（調査で確定）**: 現行は「月×行」でカード生成するため、同じ人が翌月も不能だと月別に二重カードになり（実測: マチダ 5月+6月／イグサ 4枚）、同月の複数サービス行も別カード化する。これを「**アクティブは人ごと常に1枚**」へ組み替える。

- **breakdownモデル**: レコードに `breakdown: [{month, amount}]` を持たせ、月別内訳＋合計で1枚に集約。`fnkNormalizeRecord` が既存レコードに `breakdown=[{month, amount}]`（自分自身1要素）を自己補完（後方互換）。カード表示は「4月分¥5,840／5月分¥2,920／合計¥8,760」。
- **同月複数行の合算**: 取込時、同月・同customerIdの複数不能行は**金額合算して1カード**（breakdown 1要素・amount=月合計）。
- **自動消込（fnkGoushanCandidates 昇格）**: 取込時、過去未回収 × 今月成功行出現 → `status='回収済'/resolvedMonth=当月/collectMethod='翌月合算'` を**自動確定**（承認パネル廃止）。取込サマリに「**自動消込N件: 名前¥額**」を明示。取消は既存 `fnkUndoCollected` で担保。
- **繰越**: 過去未回収 × 今月も不能行出現 → **前月カードを `status='繰越'` でクローズ**、当月に新カード（`breakdown = 前月breakdown + [{当月, 当月分}]`）。**当月分 = 電算累積額 − 前月未収合計**（実データ: マチダ 9,476−4,753=4,723）。`occurrence = breakdown.length`／`prevAmount = 前月未収合計`（既存バッジ連動）。**連絡状態はリセット**（`contactedBy/At/Method=null`・`status='未対応'`）＝新しい当月分は未対応から。前月の連絡記録は**繰越カードとGAS履歴に残る**（履歴は消さない）。`fnkIsUnpaid` に `status!=='繰越'` を追加（既存 '回収済' 除外と同型）。
- **要確認**: 取込時、今月ファイルの全customerId（成功＋不能）に過去未回収顧客が**不在** → `nextMonthAbsent=true`（自動で消さず「⚠️翌月請求なし・要確認」で残す）。
- **表示（月フィルタ独立）**: **未回収セクションは月ドロップダウンから独立**し、常に全アクティブカード（人ごと1枚）を出す。月ドロップダウンは**回収済/繰越/履歴の閲覧用**として維持。
- **冪等**: 同月再取込で壊れない（当月カードが既にあれば新規生成せず・繰越/消込を二重適用しない）。

## 4. 連絡済み記録（genba Phase 2 パターン流用）

- カードに「連絡済み」ボタン → スタッフ名ピッカー（genbaと同じ職員リスト）→ 手段選択（電話/口頭/家族経由）
- 記録 = {誰が・いつ・手段} を GASシートに**追記型（additive・上書きしない）**
- 画面即反映: カードにバッジ「✓連絡済 7/10 下浦」
- 状態は3段: **未対応 → 連絡済み → 消込**。連絡済みと消込は別（連絡したがまだ回収前、が通常）

## 6. 実装前の事前調査（クロコ・調査のみ・変更禁止）

実装GOの前に以下を一次情報で確定させる。**全部クリアするまで実装しない。**

1. **電算の合算仕様**: 合算分が翌月も不能になった場合、不能行にどう出るか（金額2ヶ月分の1行? 行が分かれる?）。「翌月不能リストに居ない＝成功」の断定可否がここで決まる。実データ（過去のkekka.xls）で確認。
   - **✅調査済み（2026-07-10・4ヶ月連続実データ 20260327/0427/0527/0629_kekka.xls）**:
     - 翌月成功時＝**前月不能顧客は翌月ファイルに成功行(code=0)として合算額で出現**（消えるのでなく積極シグナルが立つ）。決定例: cid=149 ﾑﾗﾀｷﾐｴ 3月不能2行 ¥5840+¥2920=¥8760 → 4月に成功行¥8760（金額ピタリ一致で合算確認）。
     - 翌月も不能時＝**顧客番号あたり1行に累積合算**（3月2行だった cid=151 が4月以降1行 ¥3302+¥741→¥5863→¥9030→6月成功¥11591）。行は分かれない・金額は前月未収＋当月分の累積。
     - **「不能リストに不在」単独での成功断定は危険**: 月により請求対象が大きく変動し（3月208行→4月110行）多数の顧客が翌月ファイルから消える。「成功で消えた」と「請求対象外で消えた」を区別できない。→ §3の自動消込は**成功行への出現**で判定する（消極判定を積極判定に補強済み）。
2. **DLフォルダ走査**: クロコから `C:\Users\mh\Downloads` の読取・`処理済み\` への移動が権限的に可能か実測。
3. **genba Phase 2 コードの流用可否**: recordPastContact まわりの実装を読み、furikae用に流用できる範囲を特定。
4. **現行furikae.htmlとGASシートの構造**: カード状態（未対応/連絡済み/消込/2回目）と連絡記録を持たせるためのシート拡張案。既存の消込・冪等ロジックとの整合。
5. **伝達ボード通知の除去範囲**: 「未回収◯件」通知のコード位置と、外した場合の影響範囲（morningDigestの未回収1行は残すので混同しない）。

## 7. 実装順序（調査クリア後の想定）

1. **③カード機能**（やること文面・連絡済み記録・手動消込・2回目昇格）— シート拡張含む

## 8. 将来構想: お金の見える化台帳（②③のスコープ外・実装しない）

社長要望（2026-07-10）: 利用者ごとに「利用月→請求月→国保連請求額→自己負担額→その他自費→振替結果（済/不能/請求不能）」まで、お金の流れを1本で把握できる台帳が欲しい。

- 位置づけ: 不能トラッカー＝**例外管理**（子）、台帳＝**全体の売掛台帳**（親）。台帳の「振替不能」セルの明細が不能トラッカーのカード。
- データ源は確認済み: **電算＝自己負担分の一覧出力可・けあ蔵＝国保連請求の一覧出力可**（詳細はクロコが把握）。振替結果＝kekka.xls（②で取込済みになる）。手入力ゼロで組める見込み。
- 月次フロー想定: 電算・けあ蔵から一覧出力 → 取込（②の導線を2ファイルに拡張）→ 台帳に当月行が生える → kekka.xls取込が振替結果列を埋める → 不能は③のカードへ。
- **②③実装への影響は1点のみ**: 取込時にデータを捨てない。不能行だけでなく、対象月・総件数・全行の軽いサマリ（顧客番号・金額・結果コード）を後から台帳が参照できる形で保存しておく。それ以外は②③の設計を変えない。
- 着手は②③完成後。設計セッションを別途行う。
